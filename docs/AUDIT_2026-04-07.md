## Mudanças aplicadas

### 1. Tier routing por complexity

`src/router.py::_apply_claude_tier()` faz downgrade automático de qualquer variante Claude conforme `task.complexity`:

| Complexity | Antes | Depois |
|---|---|---|
| 1-2 (low) | claude-opus-4-20250514 | **claude-haiku-4-5** ($0,80/$4 — 19x mais barato) |
| 3 (medium) | claude-opus-4-20250514 | **claude-sonnet-4-5** ($3/$15 — 5x mais barato) |
| 4-5 (high) | claude-opus-4-20250514 | **claude-opus-4-6** (modelo atual, era versão antiga) |

Mesmo padrão aplicado para Perplexity (`_apply_perplexity_tier`):
- Complexity 1-3 → `sonar-pro`
- Complexity 4-5 → `sonar-deep-research` (raciocínio multi-step)

### 2. Cap 80% por provider

`src/router.py::_apply_concentration_cap()` rebalance automático: após 5+ tarefas executadas, se algum provider passar de 80% do uso na sessão, o router escolhe a primeira alternativa viável de outro provider na chain de fallback. Configurável via env: `CAP_ANTHROPIC_SHARE`, `CAP_OPENAI_SHARE`, etc.

**Validação**: 22 tarefas simuladas pós-fix → Anthropic 45% (era 96,7%), OpenAI 18%, Perplexity 23%, Google 9%, Groq 5%.

### 4. FinOps tracker religado

`output/.finops/daily_spend.json` parou de ser populado em 30/03 porque o `cli.py:_execute_plan` (path legacy usado pelo CLI) **nunca chamava `finops.record_cost()`** — só o `Pipeline class` (path novo, não invocado pelo CLI) tinha essa integração.

**Fix**: adicionado `from src.finops import get_finops` + `finops.pre_execution_check(...)` antes do plano + `finops.record_cost(...)` após cada `agent.execute()` que retornou success.

`_resolve_provider` agora aceita 3 formatos: nome de provider (`anthropic`), nome interno (`claude`), e **model id real** (`claude-opus-4-6`, `gemini-2.5-pro`, `sonar-deep-research`, etc.).
