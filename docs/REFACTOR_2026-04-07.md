# Refactor 2026-04-07 — CLI religado ao Orchestrator v2.0

**Branch**: `refactor/cli-orchestrator-v2`
**Escopo**: refatoracao atomica do `cli.py` para eliminar o caminho legacy
v1.0 e expor a infraestrutura v2.0 que ja existia mas estava dead code.
**Smoke test**: 20/20 tests passando, todos os comandos do CLI registrados,
import limpo, `status` ja surfaceia alerta FinOps.

## Motivacao

A auditoria executada em 2026-04-07 (resposta a uma execucao do
`geo orchestrator run` que terminou com um diagnostico preocupante do agente
de execucao) confirmou tres problemas severos:

1. **Dois caminhos paralelos no codigo**:
   - `Orchestrator.run()` (v2.0): SmartRouter + cap 80% + quality gates +
     semantic cache + code-first + checkpoint + fallback chain estruturada
     + FinOps por tarefa.
   - `cli.py:_execute_plan()` (v1.0): instancia agents diretamente, sem
     nenhuma das features acima.
   
   O `cli.py run` chamava o caminho v1.0. Toda a v2.0 era dead code no fluxo
   do usuario.

2. **Concentracao de carga em Claude**: a execucao 20260407_180740 mostrou
   12/12 tarefas indo para `ArchitectAgent` (Claude). Gasto Anthropic do dia
   chegou a US$ 4.969 / US$ 5.00 (99.4% do teto local). Cap 80% nao rodou
   porque vivia no caminho v2.0.

3. **Documentacao falsa**: `CLAUDE.md` descrevia a v2.0 como ativa ha um mes,
   quando na pratica o CLI continuava no legacy.

## O que mudou

### `cli.py` — refatoracao atomica

| Antes | Depois |
|---|---|
| `from src.agents.base import TaskResult, ...` | `from src.models import ExecutionReport, Plan, TaskResult` |
| `from src.templates.decomposition import DECOMPOSITION_PROMPT` | `from src.orchestrator import Orchestrator` |
| `from src.agents.researcher import ResearcherAgent` (e 4 outros agents) | `from src.pipeline import Pipeline; from src.smart_router import SmartRouter` |
| `MODELS = {...}` (9 entradas hardcoded) | `from src.config import LLM_CONFIGS` |
| `_select_claude_by_complexity` / `_select_groq_by_complexity` | removidos (tier routing vive no Router) |
| `_get_httpx_client`, `_create_agent`, `_decompose_demand`, `_execute_plan` | **removidos** (~290 linhas legacy) |
| `_display_plan(dict)`, `_display_summary(list[TaskResult])`, `_save_report(plan, results)` | reescritos para `Plan` / `ExecutionReport` (v2.0 shapes) |
| `run()` chamava `_decompose_demand` + `_execute_plan` | `run()` chama `Orchestrator(smart=True).run()` |
| Sem comando `resume` | `cli.py resume [-c <checkpoint>]` chama `Pipeline.resume()` |
| `status` itera `MODELS` legacy | `status` itera `LLM_CONFIGS` + alerta FinOps |
| `models` itera `MODELS` legacy | `models` itera `LLM_CONFIGS` |

### Tamanho

- **Antes**: 886 linhas (`cli.py`), com 290+ linhas de legacy.
- **Depois**: 759 linhas, ~127 linhas removidas e ~150 reescritas em torno
  dos tipos canonicos da v2.0.

### Novos flags do `run`

- `--force` — bypass do `BUDGET_LIMIT` (US$ 5.00 por execucao por padrao)
- `--no-smart` — desliga o SmartRouter (debug; volta ao Router classico)
- `--dry-run` — agora usa `Orchestrator.decompose()` e salva o `Plan`
  serializado em `output/plan_<ts>.json`

### Comando `resume` (novo)

```bash
python cli.py resume                                    # usa output/.checkpoint.json
python cli.py resume -c output/checkpoints/ckpt.json    # caminho explicito
python cli.py resume --no-smart                         # debug
```

Internamente cria `SmartRouter()` (ou `Router()` se `--no-smart`) e chama
`Pipeline.resume(checkpoint_path, router)`. O checkpoint e gravado pelo
proprio Pipeline a cada wave em `output/.checkpoint.json`. Em caso de crash,
timeout, Ctrl-C ou queda de provider, basta `python cli.py resume` para
continuar do ultimo wave salvo sem reexecutar tarefas concluidas.

### `status` agora surfaceia alerta FinOps

Alem da tabela canonica de LLMs (via `LLM_CONFIGS`), o comando `status`
agora le `FinOps.daily_status()` e imprime aviso quando algum provider
passa de 80% do limite diario. Isso teria evitado a execucao 20260407_180740
ter ido toda para Claude — o operador veria `BLOQUEADO anthropic` antes de
disparar o run.

### Schema do `output/execution_*.json` mudou

Antes (v1.0):
```json
{
  "plan": {"demand_summary": "...", "tasks": [...], "execution_plan": {...}},
  "results": [{"task_id": "...", "agent_name": "...", "model_used": "...", "cost_usd": ...}],
  "totals": {"cost_usd": ..., "duration_seconds": ...}
}
```

Agora (v2.0):
```json
{
  "demand": "...",
  "summary": "...",
  "totals": {
    "cost_usd": ..., "estimated_cost_usd": ..., "duration_ms": ...,
    "tasks_completed": ..., "tasks_failed": ..., "tasks_cached": ...,
    "tasks_deduplicated": ..., "tasks_quality_retried": ...,
    "budget_limit": ...
  },
  "plan": <Plan.model_dump()>,
  "results": {"<task_id>": <TaskResult.model_dump()>}
}
```

Scripts externos que consomem o formato antigo (`scripts/run_5llm_board.py`,
dashboards, geo-finops sync) precisam ser ajustados — backlog em **Roadmap**
abaixo.

## Como usar pos-refatoracao

```bash
# Run completo (caminho v2.0 com todas as protecoes)
python cli.py run "criar helper Python para validar acentos em URLs"

# Dry-run (so decomposicao, sem custo de execucao)
python cli.py run "<demanda>" --dry-run

# Forcar bypass do budget guard
python cli.py run "<demanda>" --force

# Retomar execucao interrompida
python cli.py resume

# Status canonico + alerta FinOps
python cli.py status

# Listar 5 LLMs canonicos
python cli.py models
```
